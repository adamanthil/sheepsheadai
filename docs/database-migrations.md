# Database Migration Workflow

We manage the Postgres schema with [graphile-migrate](https://github.com/graphile/migrate).
This document covers the day-to-day workflow: writing new migrations, deploying
to production, and recovering from common pitfalls.

All commands run from `db/` (or use `npm --prefix db run …` from the repo root).

graphile-migrate reads `DATABASE_URL`, `SHADOW_DATABASE_URL`, and
`ROOT_DATABASE_URL` directly from the environment. Make sure they are exported
before running any command — typically:

```bash
set -a && source .env && set +a
```

## Day-to-day: writing a new migration

1. **Edit `db/migrations/current.sql`.** This is the one in-progress migration.
   Treat it as a scratchpad — `gm watch` (next step) will keep applying it on
   save, so write it idempotently (`DROP … IF EXISTS` followed by `CREATE …`,
   or wrap in a transaction that rebuilds from a known state).
2. **`npm --prefix db run watch`** in another terminal. On each save,
   graphile-migrate:
   - resets the **shadow DB** to the state of all *committed* migrations +
     `afterReset.sql`,
   - applies `current.sql` to the shadow DB,
   - then applies `current.sql` to your real dev DB.

   This gives you instant feedback — a syntax error or FK violation surfaces
   immediately.
3. **Iterate** until `current.sql` is what you want.
4. **Commit the migration:**
   ```bash
   npm --prefix db run commit -- -m "add_player_streaks"
   ```
   This freezes `current.sql` into
   `db/migrations/committed/000002-add_player_streaks.sql` (hashed,
   sequenced) and resets `current.sql` to empty. **The committed file is
   immutable** — never edit it after this point.
5. **Commit to git:** the new committed migration *and* any `current.sql`
   changes (it'll be empty/blank again).

## Updating production

```bash
DATABASE_URL=... npm --prefix db run migrate
```

`gm migrate` only applies committed migrations whose hash isn't already
recorded in the `graphile_migrate.migrations` table on the target DB. It does
not need shadow / root URLs in production — only `DATABASE_URL`.

Run it as part of your deploy step, **before** the new server code starts.

## Command reference

| Command | When to use |
|---|---|
| `npm --prefix db run watch` | Active development on `current.sql`. |
| `npm --prefix db run migrate` | Apply outstanding committed migrations to the DB pointed at by `DATABASE_URL`. Idempotent — safe to run repeatedly. |
| `npm --prefix db run commit -- -m "msg"` | Freeze `current.sql` into a numbered, hashed committed migration. |
| `npm --prefix db run uncommit` | Move the most recent committed migration *back* into `current.sql`. Only safe if nobody else has applied it yet (no PR merged, no prod run). |
| `npm --prefix db run reset` | Drop and recreate the dev DB, then re-run all committed migrations + `afterReset.sql`. Useful to verify your migration chain still applies cleanly from scratch. |

## Editing committed migrations (when you really must)

Once a committed migration is on `master` or has been applied anywhere
downstream, **don't edit it** — write a new migration instead. Editing changes
the hash, and any DB that already ran the old version will refuse the new one
(or worse, ignore it silently).

If you need to undo something during local development *before merging*, use
`npm --prefix db run uncommit`.

## Reference data (`afterReset.sql`)

`db/fixtures/afterReset.sql` runs only on `gm reset` and on every shadow
rebuild during `gm watch`. It is **not** run by `gm migrate` against
production. So:

- Use it for **dev convenience** (seed `suit` / `card` so the shadow DB matches
  reality after a reset).
- For **production** reference data, write a regular migration that does the
  seeding (or have your application bootstrap it on startup, like the
  `ai_model` row in Phase 5 of the web server update plan).

Right now `afterReset.sql` seeds `suit` and `card`. When we ship to prod, the
very first deploy needs those rows too — either fold the seeding into a
one-time committed migration, or run
`psql -f db/fixtures/afterReset.sql` once after `gm migrate`.

The seed file itself is generated by `scripts/gen_card_seed.py` from
`sheepshead.py` constants. Re-run the generator (and commit the result)
whenever you change `DECK` / `SUIT_NAMES` / `CARD_FULL_NAMES`.

## CI sanity check

In CI: spin up a Postgres service, run `npm --prefix db run migrate`, then
check that `git status db/migrations/committed/` is clean. That catches the
case where someone modified a committed file by hand.

## Common pitfalls

- **Don't edit a committed migration.** Make a new one.
- **`current.sql` must be idempotent during watch.** If you `CREATE TABLE
  foo`, also handle "table already exists" — usually by `DROP TABLE IF EXISTS
  foo CASCADE;` at the top, or wrap the whole file in `BEGIN; … COMMIT;` and
  design statements to be re-runnable.
- **Production never gets `gm reset`, `gm watch`, or a shadow DB.** Only
  `gm migrate`. Do not ship `SHADOW_DATABASE_URL` / `ROOT_DATABASE_URL` to
  production env.
- **Env vars are read directly** (no `$VAR` substitution in `.gmrc`). Make
  sure `DATABASE_URL` is exported in the shell that runs the command —
  `set -a && source .env && set +a` is the easy path.
- **Hash drift**: if `npm run migrate` complains that a committed migration's
  hash doesn't match what's recorded in `graphile_migrate.migrations`,
  somebody edited a committed file. Either revert the edit or, in dev only,
  `npm --prefix db run reset`.
- **Port conflicts**: the dev Docker container exposes Postgres on
  `localhost:5433` (not 5432) to avoid colliding with a host Postgres. Make
  sure your `.env` URLs use the right port.

## Example: adding a column

```bash
# In db/migrations/current.sql:
#   ALTER TABLE player ADD COLUMN IF NOT EXISTS email TEXT;

npm --prefix db run watch          # leave running; shadow DB validates each save
# … iterate …

npm --prefix db run commit -- -m "player_email"
git add db/migrations/committed/ db/migrations/current.sql
git commit -m "DB: add player.email"

# On deploy:
DATABASE_URL=postgres://prod… npm --prefix db run migrate
```

That's the whole loop.
